# Calendar System Simplification: When Optimization Creates Overhead

*Hacker News post · October 31, 2027*

---

Productivity systems tend to accumulate. Each addition was made for a reason. The aggregate cost of all additions becomes invisible — it's distributed across small per-interaction overhead that doesn't trigger the same cost-benefit analysis that major changes do.

Calendar systems accumulate the same way. Here's a structured approach to identifying and removing the overhead.

## The overhead identification test

For each element of your calendar system (color categories, calendar layers, recurring event types, review rituals), ask: "If I removed this tomorrow, what specific problem would emerge within the next two weeks?"

If the answer is "I don't know / probably nothing / something vague like 'I'd lose structure,'" the element isn't load-bearing.

If the answer is specific ("I wouldn't know which meetings are focus-protected, and I'd start accepting meetings in my morning block"), the element is load-bearing.

## What typically survives this test

In my own systems: 3 colors (focus, meetings, other), 2 calendar layers (work, personal), 1 daily recurring block (morning focus), 1 weekly recurring event (planning review), Busy status on focus blocks.

That's the minimum viable calendar system for most knowledge workers. Everything else I've tried — additional color categories, separate block types for different projects, elaborate notification configurations, multiple review events — has been overhead that didn't survive the load-bearing test.

## The incremental addition problem

The reason systems over-accumulate: each addition is evaluated against its own cost, not against the aggregate overhead it contributes to. Adding a 7th color category costs nothing in isolation. The 7th color in a 7-color system adds cognitive overhead to every calendar interaction — but that overhead is distributed and therefore invisible.

## The simplification approach

Don't tweak — remove. Start from zero and add back only what fails the absence test. This is more efficient than incremental optimization because it forces you to evaluate each element against an empty baseline rather than an assumed-necessary background.

The first time I did this: dropped from 9 color categories to 3, from 4 calendar layers to 2, from 6 recurring block types to 1. Calendar became readable in 2 seconds instead of 10. Nothing broke.
