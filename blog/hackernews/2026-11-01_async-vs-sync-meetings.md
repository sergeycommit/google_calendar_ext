# A Decision Framework for Async vs. Sync Communication

*Hacker News post · November 1, 2026*

---

The default in most organizations is synchronous meetings for anything that requires coordination. This default is expensive (requires everyone's simultaneous availability), often unnecessary (the outcome doesn't require real-time interaction), and systematically over-applied.

A framework for making this decision explicitly rather than by default:

## The synchronous requirement test

A meeting is genuinely required when:

1. **The work benefits from simultaneous iteration.** Brainstorming where ideas build on other ideas in real-time. Pair programming. Collaborative document writing. The output quality depends on the simultaneity.

2. **Emotional or relational stakes require tone.** Difficult feedback, sensitive personnel issues, trust-building conversations. Text loses nuance; video preserves it.

3. **A decision needs to happen in real time.** High uncertainty + interdependence + time pressure. Everyone needs to be in the same state of information simultaneously to make the decision.

4. **The cost of asynchronous round-trips exceeds the cost of synchronous coordination.** A 3-hour async email chain to make a 5-minute decision is a coordination failure.

## When async suffices

- Status updates (→ async summary)
- Information sharing (→ doc or recording)
- Sequential decisions (→ decision doc with deadline)
- Input gathering (→ shared document or structured survey)
- Questions with straightforward answers (→ Slack or comment thread)

## The question to ask before scheduling

"Could this achieve the same outcome asynchronously?"

If yes: does the async version require substantially more calendar time in aggregate (from all participants) than a meeting would? If not, async is almost always preferable — it works across timezones, is searchable, and doesn't require simultaneous availability.

## The cultural dimension

In organizations with async-by-default culture, this decision is easy. In meeting-heavy cultures, proposing async alternatives feels like withholding participation. This is a cultural problem, not a communication problem, and requires a different intervention.
